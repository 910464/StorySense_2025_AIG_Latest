üìö Understanding the Analysis Parameters

This analysis evaluates your user story against 10 key quality parameters based on the INVEST criteria and industry best practices. Click on each parameter below to learn what it means, see examples, and understand how to improve your stories.

üë§ User-Centered

What does this mean?

A user-centered story focuses on the person who will actually use the feature, clearly identifying who they are and why they need this functionality.

‚úÖ Good Example
"As a busy working parent, I want to quickly reorder my usual groceries from my purchase history so that I can save time during my weekly shopping and spend more time with my family."

This clearly identifies the specific user type (busy working parent), their context, and their motivation (save time for family).

‚ùå Poor Example
"As a user, I want a button that calls the API endpoint to retrieve order history from the database."

This focuses on technical implementation rather than user needs and uses the generic term "user" without context.

üí° How to Improve:
  • Be specific about who the user is (role, context, characteristics)
  • Focus on the user's goal, not the system's functionality
  • Explain the user's motivation clearly in the "so that" clause
  • Avoid technical jargon and system-focused language

üîó Independent

What does this mean?

An independent story can be developed, tested, and delivered without waiting for other stories to be completed first.

‚úÖ Good Example
"As a registered user, I want to reset my password via email so that I can regain access to my account if I forget my credentials."

This can be implemented independently and provides immediate value without requiring other features.

‚ùå Poor Example
"As a user, I want to share my completed profile with friends so that they can see my information."

This depends on profile creation, friend management, and sharing features being completed first.

üí° How to Improve:
  • Avoid references to other unfinished features
  • Ensure the story provides value on its own
  • Break down large features into smaller, independent pieces
  • Consider what minimum functionality is needed for the story to be useful

üí¨ Negotiable

What does this mean?

A negotiable story focuses on the "what" and "why" rather than the "how," leaving room for discussion about implementation details.

‚úÖ Good Example
"As a customer, I want to be notified when my order status changes so that I can track my purchase progress."

This describes the need without specifying whether notifications should be email, SMS, push notifications, or in-app alerts.

‚ùå Poor Example
"As a customer, I want a blue popup window with Arial font to appear in the top-right corner showing order status updates every 30 seconds."

This dictates specific implementation details, leaving no room for better solutions or user experience improvements.

üí° How to Improve:
  • Focus on the outcome, not the implementation method
  • Avoid specifying UI elements, colors, or exact layouts
  • Leave technical decisions to the development team
  • Use acceptance criteria to define boundaries, not exact solutions

üíé Valuable

What does this mean?

A valuable story delivers clear, measurable benefit to users or the business.

‚úÖ Good Example
"As a mobile user, I want to use biometric authentication so that I can access my banking app quickly and securely without typing passwords."

Clear value: faster access, better security, improved user experience for a critical use case.

‚ùå Poor Example
"As a user, I want the logo to be 2 pixels larger so that it looks better."

The value is subjective and minimal - unlikely to significantly impact user experience or business metrics.

üí° How to Improve:
  • Clearly articulate the benefit in the "so that" clause
  • Consider both user value and business value
  • Quantify the impact when possible (time saved, errors reduced, etc.)
  • Ensure the value justifies the development effort

üìè Estimable

What does this mean?

An estimable story contains enough detail and clarity that developers can reasonably estimate the effort required.

‚úÖ Good Example
"As a customer service rep, I want to search for customers by email address so that I can quickly find their account information during support calls."

Clear scope: search functionality, specific field (email), defined use case. Developers can estimate this.

‚ùå Poor Example
"As a user, I want the system to be more intelligent and intuitive."

Too vague - "intelligent" and "intuitive" are subjective and don't define specific functionality to implement.

üí° How to Improve:
  • Provide enough context about the functionality needed
  • Define clear boundaries with acceptance criteria
  • Avoid vague terms like "better," "faster," or "more user-friendly"
  • Include relevant business rules or constraints

üì¶ Small

What does this mean?

A small story is appropriately sized to be completed within a single iteration or sprint.

‚úÖ Good Example
"As a user, I want to update my email address in my profile so that I receive notifications at my current email."

Focused on one specific action: updating email address. Can be completed in a few days.

‚ùå Poor Example
"As a user, I want a complete customer relationship management system so that I can manage all my business relationships."

This is an epic, not a story. It would take months to implement and should be broken into dozens of smaller stories.

üí° How to Improve:
  • Focus on one specific piece of functionality
  • Break large features into smaller, deliverable pieces
  • Aim for stories that can be completed in 1-3 days
  • Consider the minimum viable version of the feature

üß™ Testable

What does this mean?

A testable story has clear, verifiable criteria that define when the story is complete.

‚úÖ Good Example
"As a user, I want to receive an email confirmation within 5 minutes of placing an order so that I know my purchase was successful."

Specific, measurable criteria: email sent, within 5 minutes, after order placement. Easy to test.

‚ùå Poor Example
"As a user, I want the website to feel more responsive and modern."

"Feel more responsive" and "modern" are subjective - there's no clear way to verify when this is complete.

üí° How to Improve:
  • Use specific, measurable language
  • Define clear success criteria
  • Avoid subjective terms like "better," "nice," or "user-friendly"
  • Think about how you would demonstrate the feature working

‚úÖ Acceptance Criteria

What does this mean?

Good acceptance criteria are specific, measurable conditions that must be met for the story to be considered complete.

‚úÖ Good Example
1. User can enter email address in login form 2. System validates email format before submission 3. Invalid emails show error message "Please enter a valid email" 4. Valid emails proceed to password field 5. Email field remembers last successful login

Specific, testable conditions that clearly define what needs to be built.

‚ùå Poor Example
1. Login should work properly 2. Users should be happy with the experience 3. No bugs

Too vague - "work properly," "happy," and "no bugs" don't provide clear guidance for implementation or testing.

üí° How to Improve:
  • Write 3-7 specific, testable conditions
  • Use "Given-When-Then" format when helpful
  • Include both positive and negative test cases
  • Define error conditions and edge cases
  • Specify any business rules or constraints

üéØ Prioritized

What does this mean?

A prioritized story has a clear understanding of its importance relative to other work.

‚úÖ Good Example
Priority: High - Critical for Q1 launch Business Value: Reduces customer support calls by 40% User Impact: Affects 80% of daily active users

Clear rationale for priority based on measurable business impact and user benefit.

‚ùå Poor Example
Priority: Medium Reason: Would be nice to have

No clear justification for priority - "nice to have" doesn't help with planning or resource allocation.

üí° How to Improve:
  • Clearly state the priority level (High/Medium/Low or numerical)
  • Explain the rationale behind the priority
  • Consider business value, user impact, and technical risk
  • Relate to business goals or strategic initiatives
  • Include any time constraints or dependencies

ü§ù Collaboration & Understanding

What does this mean?

A story that promotes collaboration is written in language that all team members can understand.

‚úÖ Good Example
"As a customer service representative, I want to quickly access a customer's recent order history during phone calls so that I can provide faster, more helpful support without putting customers on hold."

Clear language that business stakeholders, developers, and testers can all understand and discuss.

‚ùå Poor Example
"As a system administrator, I want to implement a RESTful API endpoint with OAuth 2.0 authentication that queries the PostgreSQL database using indexed joins."

Too technical - business stakeholders can't meaningfully participate in discussions about this story.

üí° How to Improve:
  • Use business language, not technical jargon
  • Write for your least technical stakeholder
  • Focus on user outcomes rather than system behavior
  • Include context that helps everyone understand the need
  • Encourage questions and discussion during story review
